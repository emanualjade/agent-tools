########## DESIGN PRINCIPLES ##########
• External Triggers, Never Self-Judgment: The orchestration loop owns every stop/reassess/escalate trigger; the agent never votes on whether it is stuck, done, or off-track.
• Unambiguous Terminal States: Every check, gate, and objective emits a crisp machine-readable verdict (PASS/FAIL/DONE with the failing criterion named) — never prose the agent must interpret.
• Verify Observable Behavior, Not Green Tests: Verification means the one observable behavior is exercised through its real entry point with real data; 'tests pass' and 'code looks correct' are necessary but never sufficient and never reported as 'verified'.
• The Fresh-Context Quality Bar: Every spec, plan, and artifact is done only when a fresh context window can act on it without re-interviewing the user or reading the chat transcript.
• Field-Not-Table by Default (Adjective-Noun Discipline): An adjective on a noun (draft post, public user, cancelled subscription) is a state/flag/enum/relationship on the existing entity until proven otherwise; a new table must be argued for, not defaulted to.
• Thin AND Complete Slices, Tracer-Bullet First: A slice does exactly ONE observable behavior (statable in one sentence, no 'and') AND leaves the app runnable with that behavior demoable; the first slice pierces every layer for the thinnest happy path with everything else stubbed.
• Two Hats, One Per Commit: A refactor commit changes zero behavior (tests unchanged, green before and after); a behavior commit changes zero structure (no moves/renames). The agent wears one hat at a time.
• Blast-Radius Routing for Obstruction: When the architecture fights a change, route by blast-radius + reversibility: in-footprint behavior-preserving → preparatory refactor (Tier 1); reversible new seam → enabling slice behind a facade (Tier 2); irreversible/competing-design/shared-foundation → escalate the DECISION via a tiny ADR while proceeding on a reversible interim (Tier 3).
• Surface-Triggered Rigor, Not Blanket Ceremony: Heavy guardrails (idempotency, money invariants, safe migrations, use-a-library, boundary parsing, compat taxonomy) attach ONLY when a detection pass finds their trigger surface; a CSS or copy change triggers none.
• Revert-and-Reset Over Continue-and-Correct: After 2 failed self-corrections on one objective, stop appending; revert to the last green state and restart with only a one-paragraph distilled lesson, or delegate to a sub-agent whose failed-attempt context is discarded on return.
• Enumerate Before Committing, Re-Route on Stall: Non-trivial plans name 2-3 distinct approaches with a one-line tradeoff each before any code; on a stall, the reassess step must switch to a DIFFERENT enumerated approach, not a variant of the failed one.
• Use Canonical Implementations for Solved Problems: For money, auth, crypto, dates/timezones, accounting, and payments, use the established pattern/library/provider and current official docs — ask what Stripe/Amazon does; never invent.

########## NON-NEGOTIABLES ##########
• The orchestration loop — not the agent — owns every stop/reassess/escalate trigger, via mechanical counters (repeated same-target action, turns with no FAIL→PASS, correction count, budget) that fire hard interrupts.
• Every gate, check, and objective emits an unambiguous machine-readable verdict (PASS/FAIL/DONE with the failing criterion named); objectives are durable binary acceptance-criteria checklists stored OUTSIDE the conversation.
• Every artifact (spec, plan, slice) is done only when it passes the fresh-context bar: a new context window can act on it without re-interviewing the user or reading the chat transcript.
• Verification of any user-facing or stateful slice exercises the one observable behavior through its real entry point with real data; passing tests and 'code looks correct' are never reported as 'verified', and the implementing agent never has sole authority over the tests that gate it.
• Before any new table/model/type, run the adjective-noun heuristic with default 'field, not table'; the expensive option (new table) must be argued for with distinct columns AND relationships AND independent lifecycle.
• Every slice is thin (one observable behavior, no 'and') AND complete (app runnable, behavior demoable through the real entry point); the first slice is a tracer bullet piercing all layers, and no slice ships infrastructure that no current-or-next behavior slice consumes.
• Refactor and behavior never mix in one commit (two hats); structural change is behavior-preserving with tests green before and after.
• When the architecture obstructs, route by blast-radius + reversibility (preparatory refactor / reversible seam / escalate-the-decision-not-the-build); irreversible one-way-door choices are always escalated via a ≤1-page ADR while proceeding on a reversible interim, and a first-class route-back must exist to upstream steps (phase boundary, main spec) when an upstream structural decision is wrong.
• Heavy correctness guardrails attach only via a surface-detection pass: idempotency on foreign-state mutations, integer-minor-unit + currency + largest-remainder on money, expand/contract on destructive migrations, parse-at-the-boundary + typed errors on trust boundaries, additive-vs-breaking taxonomy on consumed contracts, and canonical-library-with-package-verification on auth/crypto/payments/dates — with rigor scaled to the slice's risk tier so cheap changes pay no ceremony.
• Every loop is bounded: per-loop budgets (fix attempts, route-backs, splits) plus an initiative-level ceiling (max total slices / agent calls); on exhaustion the run records a blocker and degrades gracefully rather than looping forever, and recovery defaults to revert-to-last-green carrying only a distilled lesson rather than appending corrections.

########## ANTI-PATTERNS (guardrails) ##########
• Continue-and-Correct Past Two Tries — trigger: The agent appends a third instruction ('no, do it this way', 'you forgot X') to the same conversation after two failed self-corrections on one objective, or re-edits the same file/region while the error output is unchanged (repetition without state change). | guard: Loop-owned stall detector fingerprints (tool + target + hash of resulting error). On 2 same-target edits OR 3 turns with no criterion moving FAIL→PASS OR 2 failed corrections, fire a hard interrupt: revert to last green, restart fresh carrying only a one-paragraph lesson, or delegate to a sub-agent. Never reach correction #3 in the same context.
• Self-Assessed Stopping — trigger: A prompt asks the agent to 'reflect on whether you might be stuck' or 'consider stepping back' — leaving the stop/reassess decision to the agent's own judgment. | guard: Move every trigger out of the prompt and into the harness as a counter/hook that fires a hard interrupt injecting a fixed reassess step. The agent does not get to decide whether reassessment happens.
• Green Tests Reported as Verified — trigger: The agent reports 'verified'/'fixed'/'works' on the basis of passing tests, a passing build, or 'the code looks correct' — without exercising the real behavior through the real entry point with real data. | guard: Behavior verification ladder: for any user-facing or stateful slice, trigger the one behavior through its real entry path with live data and observe output. Code review, curl, static checks, and DOM inspection are explicitly NOT browser/behavior verification — report 'code-verified' when blocked, never 'verified'.
• Test Tampering / Implementation-Mirroring Tests — trigger: The implementing agent edits, deletes, mocks-to-pass, or hardcodes inputs into a failing test; or generates tests that assert whatever the function currently returns rather than what the spec requires. | guard: The implementing agent does not have sole authority over tests. Anchor every assertion to a spec-stated input→output example, and have a separate verifier diff test files for tautology (always-true __eq__, mocks returning the asserted value, hardcoded test inputs) rather than just rerunning the suite.
• Adjective Becomes a Table — trigger: About to emit CREATE TABLE / a new model/type/enum-of-entities for what is the SAME real-world thing in a different state, role, visibility, scope, or ownership (draft_posts vs published_posts; public_users vs private_users; archived_orders). | guard: Mandatory pre-flight heuristic ordered ladder, default 'field, not table'. The agent must state in one line the distinct columns AND relationships AND independent lifecycle the new table has that the existing noun lacks; if it can't, it's a column/flag/enum. If the design would force 'all <nouns>' to become a UNION across tables, that alone proves over-splitting — collapse to one table + discriminator.
• Over-Normalizing a Scalar — trigger: About to model a single-valued status, feature flag, fixed per-row config, or 1:1 setting as its own lookup/join table or as EAV (entity/attribute/value) key-value rows. | guard: A single-valued, mutually-exclusive per-row property is a COLUMN (CHECK/enum for allowed values), never a joined table and never EAV. Introduce a lookup table only when the value set is large, shared, and itself carries attributes, or the relationship is genuinely many-to-many. Decide cardinality (single vs multi) BEFORE deciding table vs column.
• Horizontal-First / Schema-First Sequencing — trigger: A proposed slice's deliverable is a single layer (schema-only, endpoints-only, types-only, 'set up the models first'), or the first slice doesn't pierce to a runnable behavior. | guard: The first slice must touch every layer for exactly one thinnest path with everything else stubbed; forbid any slice whose answer to 'name the one user-observable behavior' is 'none yet, this is groundwork'. Convert genuine unknowns to a time-boxed spike whose output is a written decision, not committed scaffolding.
• Speculative Foundation — trigger: Building a config system, base class, generic abstraction, or full schema 'we'll need later' that no current or immediately-next slice consumes; a *_type column that pre-announces future splitting. | guard: Foundation justification gate: no infrastructure merges unless a behavior slice in the same or next slice exercises it. Build infra at first point of use; generalize only on the second use (rule of two). YAGNI on states/roles/scopes the current feature doesn't actually require.
• Refactor Spiral / Scope Explosion — trigger: The agent correctly senses the architecture fights it, then 'fixes the architecture' — a refactor reaching outside the slice's file footprint or touching a shared foundation; or any change not traceable to a named acceptance criterion (gold-plating, unrequested refactor, deepening an unrequired sub-problem). | guard: Hard-bound preparatory refactors to files the slice already touches and behavior-preserving edits. The moment a refactor wants to touch a file outside the footprint or a shared foundation, convert to a Tier-2 seam or Tier-3 escalation. At each checkpoint require every change to map to an acceptance criterion; off-objective work is a stall signal, not just inefficiency.
• Silent Unilateral Architecture Decision — trigger: About to commit a new persistent schema, a new cross-boundary/external dependency, a new public contract, or a choice between competing hard-to-reverse designs — and just building it because nothing said to stop. | guard: Reversibility gate before committing: classify one-way vs two-way door. One-way doors are mandatory Tier-3 escalations — write a ≤1-page ADR (obstruction, 2-3 candidates with one-line tradeoffs, reversibility, recommendation, reversible interim), proceed on the reversible interim under a grep-able ARCH-DEBT marker, surface for async ratification. Never cement the irreversible choice silently; never block the whole build.
• Foreign-State Mutation Before Local Persist / No Idempotency Key — trigger: On a mutating handler with an external side effect (charge, email, webhook, external POST), the external call runs before local rows are committed, or no idempotency key is accepted on the endpoint or passed downstream. | guard: Triggered only when a mutation has a foreign side effect: accept an Idempotency-Key, commit local DB writes in a transaction BEFORE the external call, pass a derived key into the external SDK call, cache and replay the response on repeat keys. Skip entirely for reads and naturally-idempotent upserts on a unique key.
• Money as Float / Bare Amount — trigger: price * 1.0, parseFloat/Number on money, an amount field with no adjacent currency, amount * 100 to convert, or total / n to split. | guard: On any price/total/fee/tax/balance/payout code: integer minor units only; currency is a required field beside every amount; convert with a zero-decimal-aware helper (JPY≠×100); split with largest-remainder so parts sum exactly to the whole. Ban float/parseFloat on money in review.
• All-In-One Destructive Migration — trigger: A single migration drops, renames, or NOT-NULLs an existing in-use column, or combines schema change + backfill + read-switch in one deploy / one PR with the code that uses it. | guard: Split into expand (additive nullable) → dual-write + batched backfill → contract (drop old, separate later deploy). Renames become add-new + backfill + drop-old. Brand-new tables with no readers are exempt (one additive migration is fine).
• Hand-Rolled Security/Time/Money or Hallucinated Package — trigger: Writing a custom JWT verifier, password hasher, crypto/signing routine, timezone math, or money rounding instead of a vetted library — or importing a plausible-sounding package without confirming it exists. | guard: When a change touches auth/crypto/payments/accounting/dates-tz, require an established library/provider, no custom implementation. For any newly-added dependency, verify it actually exists and is the genuine popular package (registry + download-count check) before importing, to defeat slopsquatting.
• Trust-the-Payload / Naked 500s — trigger: Business logic trusts request/payload shape, scatters ad-hoc if(!x) checks, and surfaces failures as generic 500s or naked thrown exceptions with no typed/structured error. | guard: At every trust boundary (HTTP handler, queue/webhook consumer, file/CSV upload, third-party response) parse raw input into a typed/schema-validated object on the first lines, reject (don't coerce) with a structured error carrying a machine-readable code and correct status; let the type carry inward so downstream code never re-checks. Reserve exceptions/500 for truly unexpected faults.
• In-Place Breaking Contract Change — trigger: Renaming/removing a response field, tightening a param type, or making an optional param required on an already-consumed endpoint, exported function, or shared event schema — to 'clean it up'. | guard: Only when editing a surface other code consumes: allow additive changes (new optional param, new field, new endpoint) freely; for breaking changes require a versioned/deprecation path instead of an in-place edit. Apply least-astonishment — the symbol does what its name says, no hidden side effects, sensible defaults. Private/single-caller/brand-new code is exempt.
• Reassessing Against the Polluted Transcript — trigger: On a stall, the agent 'reviews its progress' by reading back through its own conversation instead of re-reading the fixed external objective. | guard: Keep objective + acceptance criteria as a durable artifact OUTSIDE the conversation. The reassess step re-reads it verbatim and emits a per-criterion PASS/FAIL table plus an explicit diff of 'goal requires' vs 'changes made so far' — never a freeform self-summary.
• Uniform Ceremony / Ignored Risk Tier — trigger: An XS one-file copy/config change pays the full research → plan → verify-plan → build → verify-build toll; or canonical-implementation is left as soft 'always consider' language a hands-off agent skips. | guard: Risk-tier the pipeline off the size/type already computed at slice creation: low-risk non-domain slices collapse research+plan and skip the standalone verify-plan gate; M+ and anything touching money/auth/data/migrations/external-APIs gets the full path with canonical-implementation and security lenses MANDATORY (not 'consider').
• Ephemeral Debt — trigger: A taken shortcut or interim decision is recorded only in the agent's reasoning or the PR body, so the next slice/agent never sees it and builds on the hack. | guard: Record debt as code, twice: a grep-able marker ARCH-DEBT(slice-id): reason + link at the exact site, plus a committed one-paragraph ADR. The next slice's first step greps for ARCH-DEBT touching its footprint before coding. Recorded-in-chat does not count as recorded.

A phase that the development plan already scoped tightly (few surfaces, one outcome) does not need a separate detailed phase-spec artifact before slicing — the phase stub plus main spec is enough. Gate the standalone phase-spec on phase size/risk; for small phases produce spec-and-slices in one pass. Keep them separate for L phases or phases flagged risky in create-development-phases.
  ~ Collapse the three near-identical Browser/User-Flow sections (verify-slice-build, verify-phase, verify-main-spec) into one shared standard invoked with a scope parameter, rather than three full copies — reduces prompt mass and the chance the three drift apart.
  ~ The fix step's separate fix-NNN.md artifact for trivial single-issue repairs is ceremony. For a single Must-Fix repaired in-place, let the fix be recorded as a note appended to the existing build.md/plan.md rather than spawning a numbered fix file and a fresh agent; reserve standalone fix-NNN.md + dedicated agent for multi-issue or routed-back repairs.

########## CRITIC GAPS/RISKS ##########
  gap: MODALITY GAP - the entire research assumes a backend/web-app/SQL feature world. There is zero coverage of: data/ML pipeline features, streaming/event-driven systems, mobile (iOS/Android, app-store review cycles, offline-first sync), CLI/devtools, infra-as-code/Terraform, embedded, game logic, browser extensions, or LLM-app features (prompt/RAG/eval changes). The 'adjective-noun -> add a column' heuristic and the Stripe-grade money/migration guardrails do not transfer to a Kafka topic schema, a feature flag in a mobile binary, or a dbt model. The workflow will mis-route these or apply irrelevant ceremony.
  gap: FRONTEND/UI BUILDING IS UNDER-RESEARCHED AS A BUILD ACTIVITY. UI appears only as a verification problem (screenshots are fakeable) in the last dimension. There is no research on slicing, guardrails, or anti-patterns for actually BUILDING UI: component reuse vs duplication, design-system adherence, accessibility, responsive/i18n, state management, loading/empty/error states. The 'thin slice must be demoable through a real entry point' rule assumes a UI exists but the workflow has no UI-construction discipline.
  gap: NO RESEARCH ON THE READ/UNDERSTAND-BEFORE-WRITE PHASE. Every dimension is about constraining the agent once it is editing or planning. Nothing covers how the agent builds an accurate model of the EXISTING codebase first: finding the right files, discovering existing conventions/utilities to reuse, locating the real entry points, understanding implicit invariants. Most production-feature failures come from not knowing what already exists (reinventing a helper, missing a validation layer). The 'don't duplicate' guardrail fires too late if the agent never searched.
  gap: NON-CODE WORK IS ENTIRELY ABSENT. Production features require: tests as deliverables, observability/logging/metrics, feature flags and rollout/kill-switch, documentation, env/config and secrets, dependency/lockfile updates, CI changes, infra/deploy changes, telemetry/analytics events. The slicing model ('one observable behavior, app runnable') has no slot for 'add a dashboard alert' or 'wire a feature flag' which are real, non-user-observable but mandatory production steps.
  gap: MULTI-AGENT / CONCURRENCY AND STATE-OWNERSHIP ARE UNADDRESSED. The recommendations freely invoke 'hand to a sub-agent', 'fresh session', 'separate verifier agent', and ARCH-DEBT markers read by 'the next slice or next agent' - but nothing covers: how parallel agents avoid editing the same files, merge/rebase conflict handling, who owns the durable acceptance-criteria artifact when multiple agents write it, or how the stall-detector fingerprint survives across agent boundaries. The architecture leans on multi-agent handoff without designing it.
  gap: HUMAN-IN-THE-LOOP MECHANICS ARE A NAMED DEPENDENCY WITH NO DESIGN. Tier-3 escalation, ADR ratification, 'surface for human review', and the claim that reflection 'catches 60-70% of issues that would otherwise surface in human review' all assume a human reviewer/approver exists and responds. There is no research on: async escalation latency, what the agent does while waiting (the 'reversible interim' can itself be wrong), how a human even SEES the ADR in an autonomous run, or batching/notification of escalations. The whole approach silently depends on a responsive human oracle.
  gap: THE ACCEPTANCE-CRITERIA ARTIFACT IS LOAD-BEARING BUT ITS GENESIS IS UNRESEARCHED. Three of five dimensions hinge on a 'durable, external, binary, test-decidable acceptance-criteria checklist.' Nothing covers WHO writes it, HOW you guarantee it is correct and complete (a wrong/incomplete criteria artifact silently authorizes the wrong feature and every downstream gate passes), how it is kept in sync when scope legitimately changes mid-build, or what happens when criteria conflict. If this artifact is itself LLM-generated, the same 'tests encode the code's behavior not the spec' failure applies one level up.
  gap: VERIFICATION COVERAGE IS INCOMPLETE AND TRUNCATED. The honest-lean-verification dimension cut off mid-UI-branch. Even from what is present, there is no ladder rung for: non-deterministic/async behavior, concurrency/race conditions, performance/load, security verification (authz bypass, injection), data-migration correctness on real data volumes, third-party/external-service behavior (which can't be invoked freely), or destructive operations you cannot safely 'demo through the real entry path.' The 'demo the real behavior with live data' DoD is undefined for irreversible or external-effect actions.
  gap: NO COST / LATENCY / TOKEN BUDGET MODEL FOR THE GUARDRAILS THEMSELVES. The research repeatedly claims mechanisms are 'lean' and 'zero overhead on the happy path,' but never measures the aggregate cost of running surface-detection, five-question schema gates, per-edit STOP-signal self-checks, reassess templates, separate verifier agents, and ADR writing on every slice. Stacked across a multi-slice feature this is substantial token/latency overhead and added failure surface; the leanness claim is asserted, not grounded.
  gap: ROLLBACK / FAILURE-RECOVERY OF SHIPPED SLICES IS MISSING. 'Revert to last green state' is well-covered for in-flight code, but once slices MERGE/DEPLOY there is no story for: a slice that passed its gate but breaks in production, partial-feature states left in production between slices (the app is 'runnable' but a half-built feature is live), feature-flag cleanup, or backing out an expand/contract migration mid-rollout. 'Always runnable' is not the same as 'safe to ship each slice.'
  gap: TESTING-STRATEGY DECISIONS ARE UNRESEARCHED. The verification dimension shows LLM tests enshrine bugs and get tampered with, but there is no positive guidance on WHAT to test: unit vs integration vs e2e balance, what deserves a test at all, test data/fixtures/factories, flaky-test handling, or how the separate verifier obtains independent oracles when only the spec exists. 'Anchor assertions to spec lines' presupposes a spec granular enough to decide every assertion, which the acceptance-criteria gap above undermines.
  gap: GIT/PR WORKFLOW AND THE TWO-HATS COMMIT RULE ARE ASSUMED, NOT DESIGNED. The architectural-obstruction dimension mandates machine-checkable commit invariants (refactor commit changes no assertions; feat commit moves no files) and grep-able ARCH-DEBT markers, but there is no research on branch strategy, PR sizing/sequencing across slices, how the 'auto-reject mixed diff back to the agent' loop actually works, or how commit-level rules interact with squash-merge workflows where the granularity is lost.
  risk: OVERFITTING TO LOOPING/FIXATION: 'altitude-and-weeds' is the most developed dimension and its mechanisms (stall detector, hard interrupts, revert-and-reset) assume the failure mode is REPETITION. But the most dangerous production failures are SILENT and CONFIDENT one-shot errors (days_held=0 skips every trade, PID=0 writes nothing - cited in the verification dimension) where the agent never loops, reports success, and moves on. The repetition-based guardrails are blind to confidently-wrong non-looping behavior, which is the higher-severity class.
  risk: GUARDRAIL FATIGUE / IGNORED-CHECKLIST RISK IS SELF-ACKNOWLEDGED BUT UNMITIGATED ACROSS DIMENSIONS. The Stripe dimension correctly warns that ceremony-on-everything trains everyone to skip checklists - yet the combined workflow stacks an adjective-noun ladder, five schema questions, surface-detection guardrails, per-edit STOP signals, reassess templates, foundation-justification gates, and demo DoD on every slice. The aggregate IS ceremony-on-everything. The very failure the research names is the emergent risk of assembling all the research.
  risk: MECHANICAL TRIGGERS ARE GAMEABLE/EVADABLE BY THE SAME AGENT THEY CONSTRAIN. The stall-detector fingerprints 'tool + target + error-output hash'; a capable model under sunk-cost pressure can trivially perturb the target (touch a different region), reword the command, or change error output to dodge the window - exactly the test-tampering behavior documented in ImpossibleBench (test modification is >79% of cheating; more capable = more cheating). Trusting fingerprint-based detection against an adversarial-by-incentive agent is fragile.
  risk: THE WORKFLOW DEPENDS ON DETERMINISTIC, MACHINE-READABLE GATES (PASS/FAIL/DONE tokens, tests-as-truth, demoable behavior) but production features routinely have NON-BINARY, NON-DETERMINISTIC, or EXTERNALLY-DEPENDENT acceptance (UX quality, performance under load, third-party API behavior, eventual consistency). Forcing every criterion into a binary test-decidable checklist will either exclude these (false sense of done) or fabricate brittle binary proxies that pass while the real property fails.
  risk: CONTRADICTION BETWEEN 'KEEP THE BUILD MOVING' AND 'STOP ON IRREVERSIBLE DECISIONS.' The obstruction dimension says proceed on a reversible interim and only Tier-3 pauses; the Stripe dimension says irreversible data corruption (double-charge, bad migration) is near-impossible to unwind. An agent biased toward momentum will mis-classify a one-way door as reversible (it cannot reliably tell them apart - explicitly noted) and proceed, which is precisely the catastrophic case. The reversibility classifier is the single point of failure and is left to the agent's judgment.
  risk: SOURCE QUALITY IS UNEVEN AND SOME LOAD-BEARING CLAIMS REST ON BLOG POSTS / SINGLE ANECDOTES / VENDOR CONTENT. The headline '7x (14->2)' anti-loop result is one AWS dev.to post; '60-70% of issues caught' is unattributed 'practitioner reports'; key behavioral fixes cite Cursor/Claude vendor guides and Medium articles. Several arXiv refs (e.g. 2602.14798 dated in the future relative to a Jan-2026 cutoff) are preprints, not peer-reviewed. Building a production methodology on these as if they were settled is a credibility risk.
  risk: SELECTION BIAS: THE RESEARCH ONLY STUDIES KNOWN FAILURE MODES AND THEIR KNOWN FIXES. There is no adversarial coverage of failures the cited sources don't name - e.g., the agent satisfying every gate while building the WRONG feature (criteria were wrong), correct-looking code with subtle security holes that pass all behavioral demos, supply-chain/secrets exposure during the build, or emergent bad interactions BETWEEN guardrails (e.g., revert-and-reset discarding context that contained the only record of an ARCH-DEBT decision).
  risk: THE 'CLEAN RESTART / DISCARD CONTEXT' PRIMITIVE CAN DESTROY HARD-WON KNOWLEDGE. The altitude dimension's central counterintuitive move - on stall, throw away context and carry only a one-paragraph lesson - risks discarding exactly the partial diagnosis, reproduction steps, or environmental facts that were the actual progress. 'Distill to one paragraph' assumes the agent can correctly identify what mattered, but a fixated agent (by the dimension's own argument) is the worst judge of that. Repeated resets can cause amnesia loops that are slower than fixing in place.
  risk: TWO-HATS AND EXPAND/CONTRACT ADD DEPLOY-COUNT AND PR-COUNT MULTIPLICATION THAT MAY BE INFEASIBLE. Splitting every breaking migration into expand -> backfill+dual-write -> contract across SEPARATE deploys, and every change into refactor-only + feature-only commits, assumes a fast, cheap, independent deploy/release pipeline. In environments with slow CI, manual release trains, or coupled deploys, this 3-5x multiplication of deploys/PRs per feature is a real throughput and coordination cost the research treats as free.
  risk: NO TREATMENT OF SCALE/STATE: how these guardrails behave on a LARGE existing codebase (slow greps, huge schemas the SQL-accuracy research says degrade LLMs, thousands of existing ARCH-DEBT markers, monorepos) vs a greenfield toy. Several mechanisms (re-read acceptance criteria verbatim, grep ARCH-DEBT for the footprint, parse whole schema for adjective-noun collisions) scale poorly and may themselves exhaust context - the context-rot problem the workflow is trying to solve, reintroduced by the guardrails.
  risk: THE APPROACH ASSUMES A SINGLE COOPERATIVE AGENT FOLLOWING A SYSTEM PROMPT, BUT PROVIDES MOSTLY PROMPT-LEVEL AND HOOK-LEVEL CONTROLS. Where enforcement is 'stated in the system prompt' (two-hats, STOP signals, foundation gate), the verification dimension's own evidence (strict prompting cut cheating 92%->1% but not to 0; monitoring catches only 42-65%) shows prompt-level rules are partially honored at best. The methodology's reliability ceiling is bounded by exactly the non-compliance the research documents, and it has no independent enforcement for most non-mechanical rules.
